MODEL: Usuario (Custom User)
Local: apps/accounts/models.py

Campos principais:
- Herda de AbstractUser do Django
- telefone (CharField, max=20)
- cpf (CharField, unique=True, max=14) - Com validação
- curso (CharField, max=100)
- matricula (CharField, unique=True)
- is_staff_member (BooleanField) - Para scanner de validação
- is_admin (BooleanField) - Para aprovação de documentos

Método cpf_valido():

- TrocarSenhaSerializer: Alteração de senha com validação

VIEWS: apps/accounts/views.py
- CustomTokenObtainPairView: Login JWT customizado
  * Retorna access token, refresh token e dados do usuário
- RegisterView: Cadastro de novo usuário

- TrocarSenhaView: Alterar senha do usuário

🛒 MÓDULO: orders (Pedidos e Documentos)
================================================================================

MODEL: Pedido
Local: apps/orders/models.py

Campos:
- usuario (ForeignKey para Usuario)
- evento (ForeignKey para Evento)
- status (Choice):
  * PENDENTE_VALIDACAO: Aguarda aprovação documento
  * APROVADO: Aprovado, pode pagar
  * REJEITADO: Documento rejeitado
  * PAGO: Pago com sucesso
  * CANCELADO: Cancelado
- valor_total (DecimalField)
- criado_em, atualizado_em (TimestampedModel)

Propriedades:
- total_ingressos: Soma de itens
- requer_documento: Verifica se algum lote requer
- documento_pendente: Verifica se documento está pendente

MODEL: ItemPedido
Local: apps/orders/models.py

Campos:
- pedido (ForeignKey para Pedido)
- lote (ForeignKey para Lote)
- quantidade (PositiveIntegerField)
- preco_unitario (DecimalField - salvo automaticamente do lote)

Propriedade:
- subtotal: quantidade × preco_unitario

MODEL: Documento
Local: apps/orders/models.py

Campos:
- pedido (OneToOneField para Pedido)
- tipo (Choice): ATESTADO_MATRICULA
- arquivo (FileField) - Upload PDF
- status (Choice):
  * PENDENTE: Aguarda verificação
  * APROVADO: Verificado e aprovado
  * REJEITADO: Rejeitado
- verificado (BooleanField)
- motivo_rejeicao (TextField)
- verificado_por (ForeignKey para Usuario - Admin)
- verificado_em (DateTimeField)

SERIALIZERS: apps/orders/serializers.py
- DocumentoSerializer: Serialização completa
- DocumentoUploadSerializer: Upload de PDF
- ItemPedidoSerializer: Com detalhes do lote
- PedidoSerializer: Lista de pedidos
- PedidoDetailSerializer: Detalhes com documento
- CheckoutSerializer: Validação do checkout
- AdminVerificacaoSerializer: Aprovação/rejeição

VIEWS: apps/orders/views.py

PedidoViewSet (ModelViewSet):

Fluxo de Checkout (create):
1. Recebe evento_id e itens (lista de lote_id + quantidade)
2. Verifica se evento existe e está ativo
3. Verifica se evento ainda não ocorreu
4. VERIFICA LIMITE: Máximo 2 ingressos por pessoa/evento
   - Soma pedidos existentes (PENDENTE/APROVADO/PAGO)
   - Soma novos itens
   - Retorna erro se exceder 2
5. Para cada item:
   - Verifica disponibilidade do lote
   - Verifica estoque suficiente
   - Marca se requer documento (CALOURO/VETERANO)
   - Cria ItemPedido
   - Decrementa quantidade_disponivel do lote
6. Calcula valor_total
7. Se NÃO requer documento → status = APROVADO
8. Se requer documento → status = PENDENTE_VALIDACAO
9. Retorna dados do pedido criado

Action: cancelar
- Verifica se usuário é dono ou admin
- Não permite cancelar se já pago
- Restaura estoque dos lotes
- Atualiza status para CANCELADO

DocumentoUploadView (APIView):
- Recebe pedido_id e arquivo PDF
- Verifica se pedido existe e pertence ao usuário
- Verifica se pedido está PENDENTE_VALIDACAO
- Verifica se requer documento
- Upload ou atualização do arquivo

AdminVerificacaoViewSet (ModelViewSet):
- pendentes: Lista documentos não verificados
- partial_update (PATCH): Aprova ou rejeita
  * Se APROVADO:
    - Atualiza documento e pedido
    - pedido.status = APROVADO
  * Se REJEITADO:
    - Atualiza documento com motivo
    - pedido.status = REJEITADO
    - Restaura estoque dos lotes

URLS: apps/orders/urls.py
- POST /api/pedidos/                    - Checkout
- GET  /api/pedidos/                   - Meus pedidos
- GET  /api/pedidos/{id}/              - Detalhes
- POST /api/pedidos/{id}/cancelar/     - Cancelar
- POST /api/documentos/upload/          - Upload PDF
- GET  /api/admin/verificacao/         - Listar docs
- GET  /api/admin/verificacao/pendentes/ - Pendentes
- PATCH /api/admin/verificacao/{id}/  - Aprovar/rejeitar

💳 MÓDULO: payments (Mercado Pago)
================================================================================

MODEL: Pagamento
Local: apps/payments/models.py

Campos:
- pedido (OneToOneField para Pedido)
- status (Choice):
  * PENDENTE: Aguarda pagamento
  * APROVADO: Pago com sucesso
  * RECUSADO: Recusado/falhou
  * ESTORNADO: Estornado
- metodo (Choice):
  * PIX
  * CARTAO_CREDITO
  * CARTAO_DEBITO
- payment_id (CharField) - ID do Mercado Pago
- preference_id (CharField) - ID da preferência
- external_reference (CharField) - Referência externa (pedido_id)
- valor (DecimalField)
- codigo_pix (TextField) - QR Code base64
- url_pagamento (URLField) - Link de pagamento
- aprovado_em (DateTimeField)
- webhook_received_at (DateTimeField)
- webhook_payload (JSONField) - Dados do webhook

SERIALIZERS: apps/payments/serializers.py
- PagamentoSerializer: Serialização completa
- WebhookPayloadSerializer: Validação do webhook
- CriarPagamentoSerializer: Criação de preferência

VIEWS: apps/payments/views.py

MercadoPagoPaymentView (APIView):
POST /api/payments/criar/

Fluxo:
1. Recebe pedido_id e metodo (opcional)
2. Verifica se pedido existe e pertence ao usuário
3. Verifica se pedido está APROVADO
4. Verifica se já não foi pago
5. Inicializa SDK Mercado Pago
6. Cria lista de itens da preferência
7. Adiciona dados do pagador (nome, email)
8. Configura métodos de pagamento (exclui ticket/atm)
9. Define notification_url para webhook
10. Cria preferência via API Mercado Pago
11. Cria registro Pagamento
12. Retorna: preference_id, init_point, sandbox_init_point, qr_code, qr_code_base64

MercadoPagoWebhookView (APIView):
POST /api/payments/webhook/

Fluxo:
1. Recebe payload do Mercado Pago
2. Valida estrutura do payload
3. Identifica tipo de notificação
4. Extrai payment_id ou order_id
5. Consulta API Mercado Pago para obter dados atualizados
6. Se status == 'approved':
   - Atualiza Pagamento para APROVADO
   - Define aprovado_em e webhook_received_at
   - Salva payload completo
   - Atualiza Pedido para PAGO
   - DISPARA TASK CELERY: gerar_ingressos_e_enviar_email
7. Se status in ['rejected', 'cancelled', 'refunded']:
   - Atualiza Pagamento para RECUSADO

PagamentoDetailView (APIView):
- Retorna detalhes do pagamento
- Verifica permissão (dono ou admin)

URLS: apps/payments/urls.py
- POST /api/payments/criar/      - Criar pagamento
- POST /api/payments/webhook/    - Webhook Mercado Pago
- GET  /api/payments/{id}/       - Detalhes pagamento

🎟️ MÓDULO: tickets (Ingressos e Scanner)
================================================================================

MODEL: Ingresso
Local: apps/tickets/models.py

Campos:
- uuid (UUIDField) - Identificador único, gerado por uuid.uuid4()
- usuario (ForeignKey para Usuario)
- evento (ForeignKey para Evento)
- tipo (Choice TipoIngresso)
- status (Choice):
  * VALIDO: Pode ser utilizado
  * UTILIZADO: Já foi usado na porta
  * CANCELADO: Cancelado
  * EXPIRADO: Evento já ocorreu
- qr_code_image (ImageField) - QR Code salvo
- utilizado_em (DateTimeField) - Data/hora da validação
- validado_por (ForeignKey para Usuario - staff)

Métodos:
- marcar_utilizado(staff_user):
  * Verifica se não está utilizado/cancelado
  * Atualiza status para UTILIZADO
  * Define utilizado_em com timezone atual
  * Define validado_por
- cancelar(): Cancela ingresso (se não utilizado)

TASKS CELERY: apps/tickets/tasks.py

Task: gerar_ingressos_e_enviar_email
Objetivo: Gerar ingressos e enviar email após confirmação de pagamento

Fluxo:
1. Recebe pedido_id
2. Busca pedido com usuário, evento e itens
3. Verifica se pedido está PAGO
4. Para cada ItemPedido:
   - Para cada quantidade (1 ingresso por vez):
     * Cria Ingresso com UUID único
     * GERA QR CODE com qrcode library
     * Salva imagem do QR Code no ingresso
     * Dispara task: enviar_email_ingresso
5. Retorna IDs dos ingressos gerados

Configurações:
- bind=True: Acesso à instância da task
- max_retries=3: Tenta até 3 vezes
- default_retry_delay=60: Espera 60s entre tentativas
- autoretry_for=(Exception,): Reprocessa em caso de erro

Task: enviar_email_ingresso
Objetivo: Enviar email com QR Code em anexo

Fluxo:
1. Recebe ingresso_id
2. Busca ingresso com usuário e evento
3. Prepara template context
4. Renderiza templates HTML e TXT
5. Cria EmailMessage com QR Code em anexo
6. Envia email
7. Retorna confirmação

Task: enviar_email_confirmacao_pagamento
Objetivo: Notificar usuário sobre pagamento confirmado

SERIALIZERS: apps/tickets/serializers.py
- IngressoSerializer: Serialização completa com URL do QR Code
- ScannerValidacaoSerializer: Validação de UUID para scanner

VIEWS: apps/tickets/views.py

Função: gerar_qr_code
- Gera QR Code usando qrcode library
- Parâmetros: version=1, ERROR_CORRECT_H, box_size=10, border=4
- Retorna io.BytesIO com imagem PNG

IngressoViewSet (ReadOnlyModelViewSet):
- list, retrieve: Lista ingressos do usuário ou todos (admin)
- gerar_e_salvar_qr_code: Método estático para gerar QR Code

StaffScannerView (APIView):
POST /api/scanner/validar/

Fluxo de Validação:
1. Recebe uuid no body da requisição
2. Verifica se usuário é Staff ou Admin
3. Busca ingresso pelo UUID
4. VALIDAÇÕES:
   - Ingresso não encontrado → 404
   - Status CANCELADO → 400
   - Status EXPIRADO → 400
   - Evento já ocorreu → 400
   - Status UTILIZADO → 400 com detalhes da utilização anterior
5. Se válido:
   - Chama ingresso.marcar_utilizado(request.user)
   - Retorna sucesso com dados do ingresso
6. Retorna resposta com valido, erro, ingresso_uuid, evento, tipo, usuario, utilizado_em

download_qr_code (APIView):
GET /api/ingressos/{uuid}/qr/
- Busca ingresso pelo UUID
- Verifica permissão (dono ou admin)
- Gera QR Code se não existir
- Retorna URL do QR Code

verificar_qr_code (APIView):
GET /api/ingressos/{uuid}/verificar/
- Público (AllowAny)
- Verifica status do ingresso
- Retorna informações básicas

URLS: apps/tickets/urls.py
- GET  /api/ingressos/                              - Meus ingressos
- GET  /api/ingressos/{pk}/                        - Detalhes
- GET  /api/ingressos/{uuid}/qr/                   - Download QR
- GET  /api/ingressos/{uuid}/verificar/            - Verificar
- POST /api/scanner/validar/                       - Validar (Staff)
